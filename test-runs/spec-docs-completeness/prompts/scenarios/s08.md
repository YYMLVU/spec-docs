# S08 - Adopt Legacy Project

## Prompt

This scenario uses either:
- The user-approved real project located under `real-projects/selected/`, OR
- If no real project has been approved, use the s08 fallback fixture (a simulated legacy project with existing codebase and no `docs/spec-docs/`).

Run `spec-docs adopt` on the selected legacy project.

The key requirement: `adopt` mode is for merging a completed target architecture into current architecture after verifying code matches target rules. For a legacy project that has NO prior spec-docs setup, the correct mode is `init` (existing-implementation init), not `adopt`. But if the project is being treated as "already in rebuild with a target to adopt," the `adopt` behavior applies.

Handle accordingly: either run `adopt` if the project has a completed target architecture rebuild ready, or guide toward `init` if the project is being introduced to spec-docs for the first time.

## Checklist

Verify the adoption/init output:

- [ ] If this is a first-time spec-docs introduction, guides toward gradual adoption through `init` rather than attempting a full rewrite of documentation.
- [ ] Does not attempt to rewrite all existing docs into spec-docs format in a single pass without user confirmation.
- [ ] Known deviations between code and any target architecture are explicitly documented.
- [ ] Confidence level is stated for architecture selection and facts.
- [ ] Missing facts are not guessed -- uses `[NEEDS CLARIFICATION: ...]` markers.
- [ ] Scope is controlled: identifies what is in scope for the current run vs what would require separate sessions.
- [ ] For a true `adopt` scenario (rebuild complete, target matches code): verifies code matches target rules before merging, archives target docs after merge, marks rebuild completed.
- [ ] Adoption plan or migration scope is documented if incremental adoption is chosen.
- [ ] Does not create fake implementation facts about code that has not been read.
- [ ] Legacy code patterns that differ from modern expectations are described accurately, not "fixed" in docs.
