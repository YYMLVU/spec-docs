# S15 - Init Profile Boundary

## Prompt

Run `spec-docs init` in the s15 fixture directory. The fixture has only three source files, but it includes a public entrypoint, API behavior, and a data access boundary. This is a boundary case: file count alone should not force Minimal Existing Project.

## Checklist

Verify that the `init` output:

- [ ] Uses observable signals beyond file count.
- [ ] Identifies the public entrypoint in `src/index.js`.
- [ ] Identifies API behavior in `src/api.js`.
- [ ] Identifies data access behavior in `src/db.js`.
- [ ] Classifies as Standard Existing Project, or explicitly justifies Minimal Existing Project by explaining why one overview accurately covers the public entrypoint, API, and data boundary.
- [ ] Does not classify by file count alone.
- [ ] If Standard, creates grounded child specs only for observed areas.
- [ ] If Minimal, maps all three source files to `specs/project-overview.spec.md` and records why separate specs are unnecessary.
- [ ] Does not create unrelated empty category directories.
- [ ] Does not invent architecture beyond the observed entrypoint/API/data boundary.

Result: PASS / PASS_WITH_NOTES / FAIL / BLOCKED
