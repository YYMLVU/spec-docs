# S05 - Verify Drift and Architecture Violations

## Prompt

In the s05 fixture directory, `docs/spec-docs/` exists. The project was set up as a greenfield project with Clean / Hexagonal Architecture, Dependency inversion addon, and Adoption Mode `greenfield`.

The fixture includes planted violations:
1. The orders module spec (`docs/spec-docs/specs/orders/module.spec.md`) describes `src/domain/order.js` as returning `{ status: "accepted" }`. But in the actual code, `src/domain/order.js` returns `{ status: "submitted" }` -- the spec says "accepted" while the code returns "submitted".
2. Code in `src/domain/order.js` directly imports and calls `saveOrder` from `../infra/order-repository.js` instead of going through a repository port/adapter, violating domain-infrastructure separation.

Run `spec-docs verify`.

## Checklist

Verify the verify output:

- [ ] Reports `[FACT DRIFT]` for spec saying "accepted" while code returns "submitted".
- [ ] The `[FACT DRIFT]` finding includes: the specific spec file, the specific code file, the mismatched status value, and evidence.
- [ ] Reports `[ARCHITECTURE VIOLATION: INFRASTRUCTURE ACCESS]` or `[ARCHITECTURE VIOLATION: LAYER]` for the domain-to-infrastructure direct dependency in `order.js`.
- [ ] The architecture violation includes: observed issue, expected architecture behavior (domain must not depend on infrastructure), recommended action, evidence (specific file and line), and source documents checked.
- [ ] Includes a severity assessment for the architecture violation. If the skill treats `greenfield` as an unsupported or undefined Adoption Mode, it must explicitly indicate an Adoption Mode ambiguity or `[NEEDS CLARIFICATION]` rather than silently inventing severity rules.
- [ ] Also lists architecture docs checked (e.g., `current-architecture.md`).
- [ ] Does NOT claim `PASS` (the output should be `FAIL`).
- [ ] Output format uses the standard finding categories: `[FACT DRIFT]` and `[ARCHITECTURE VIOLATION: <subtype>]` with a severity tag (not required to be `error`).
- [ ] Does not report violations for code that follows the architecture rules correctly.
- [ ] Reports include recommended action for each finding.
- [ ] Overall result is `FAIL` with affected files listed and recommendation for next mode (`update` or `repair`).

Result: PASS / PASS_WITH_NOTES / FAIL / BLOCKED
