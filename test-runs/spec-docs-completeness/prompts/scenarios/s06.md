# S06 - Repair Without Unauthorized Code Changes

## Prompt

In the s06 fixture directory, `docs/spec-docs/` exists. There is a known contradiction: the module spec in `docs/spec-docs/specs/modules/report.spec.md` describes `renderReport` as returning HTML. But the actual implementation in `src/report.js` has `renderReport(items)` accepting an array of items and returning a string of item titles joined with newline characters. The spec contains a Broken Section indicating this discrepancy -- the code is authoritative.

Run `spec-docs repair` to align the docs with the current implementation.

Important: Do NOT grant explicit approval to modify `src/report.js`. The only task is to repair the docs.

## Checklist

Verify the repair output:

- [ ] Repairs the docs/spec contradiction: updates the report module spec to reflect that `renderReport(items)` accepts an array of items and returns item titles joined with newline characters, resolving the Broken Section that incorrectly claimed HTML output.
- [ ] Grounds the repair in re-reading the actual code in `src/report.js`, not in memory or assumptions.
- [ ] Updates frontmatter in the report spec: `verified_commit`, `verified_date`, `source_files`, `symbols`.
- [ ] Does NOT modify `src/report.js` in any way (no code changes, no reformatting, no `console.log` additions).
- [ ] Does NOT suggest modifying `src/report.js` or treating the spec as authoritative over the code.
- [ ] Preserves or strengthens architecture rules (does not weaken or relax rules).
- [ ] Does not rewrite architecture docs unless explicit user confirmation or an ADR is provided.
- [ ] Rebuilds `inventory.md` if symbol indexes or mappings changed due to the signature update.
- [ ] Runs `verify` after repair.
- [ ] Recommends running `spec-docs verify` if verify was not automatically run.
- [ ] If any other stale specs are discovered, reports them but does not repair them beyond scope without confirmation.
- [ ] Applies or recommends `spec-docs verify` after repair completes.
