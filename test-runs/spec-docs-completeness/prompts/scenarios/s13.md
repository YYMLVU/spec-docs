# S13 - Standard Existing Project Init

## Prompt

Run `spec-docs init` in the s13 fixture directory. The fixture contains a small but meaningfully structured existing application with a runtime entrypoint, route layer, domain logic, and external integration boundary.

The expected project profile is **Standard Existing Project** because the code has multiple implementation areas and enough structure to justify separate specs or spec sections beyond a single minimal overview.

## Checklist

Verify that the `init` output:

- [ ] Classifies the project as Standard Existing Project.
- [ ] Reads `src/server.js` and identifies the runtime entrypoint and routing behavior.
- [ ] Reads `src/routes/orders.js` and identifies the order route behavior.
- [ ] Reads `src/domain/order.js` and identifies order creation and total calculation.
- [ ] Reads `src/integrations/payment-gateway.js` and identifies the external payment integration boundary.
- [ ] Creates `docs/spec-docs/README.md`, `constitution.md`, `inventory.md`, and `specs/project-overview.spec.md`.
- [ ] Creates only child specs or child spec directories supported by current code evidence, such as runtime, interface/route, domain/module, or integration specs.
- [ ] Does not create unrelated empty category directories.
- [ ] Creates architecture docs only if boundaries are supported by code evidence or user confirmation.
- [ ] If architecture docs are created, they define enforceable placement and boundary rules rather than descriptive fluff.
- [ ] Includes Code-to-Spec mappings for all fixture source files.
- [ ] Includes Symbol-to-Spec mappings for `routeRequest`, `handleCreateOrder`, `createOrder`, and `capturePayment`.
- [ ] No roadmap, backlog, tasks.md, or plan.md exists anywhere in `docs/spec-docs/`.
- [ ] Full init verify passes or reports concrete findings before final init completion.
- [ ] Generated docs contain no `{{template_variables}}`, `TBD`, or `TODO`.

Result: PASS / PASS_WITH_NOTES / FAIL / BLOCKED
