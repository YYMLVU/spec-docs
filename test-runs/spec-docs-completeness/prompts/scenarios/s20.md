# S20 - Level 4 Architecture-Risk Escalation

## Prompt

In the fixture project, a small code change introduces a new direct import from `features/billing/internal` into `features/users`, bypassing the declared billing public API. The diff is small, but existing architecture rules require cross-module access through public contracts.

Run `spec-docs update` and classify the impact.

## Checklist

Verify the update output:

- [ ] Classifies the change as Level 4 even though the diff is small.
- [ ] Reports the architecture-risk signal: cross-module access bypasses the declared public contract.
- [ ] Does not silently rewrite `architecture/current-architecture.md`, `architecture/placement-rules.md`, or accepted ADRs to legalize the change.
- [ ] Recommends `place`, `repair`, `rebuild`, or `adopt` as the next mode as appropriate.
- [ ] Requires full verify before claiming architecture state is current.
- [ ] If a factual spec update is suggested, states that architecture risk remains and architecture currentness is not claimed.
- [ ] If the change was initially treated as localized, reclassifies upward to Level 4 when the cross-module boundary bypass is discovered.

Result: PASS / PASS_WITH_NOTES / FAIL / BLOCKED
