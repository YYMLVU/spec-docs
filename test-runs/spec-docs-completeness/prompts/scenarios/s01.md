# S01 - Empty Project Init

## Prompt

Run `spec-docs init` in the s01 fixture directory (the s01 fixture is an empty project with no implementation-relevant files).

Run two variants:

**Variant 1 (architecture supplied):**
The fixture directory is empty. When running `spec-docs init`, supply the following architecture choices when prompted:
- Primary Preset: Lightweight Tooling Architecture
- Addons: none
- Adoption Mode: greenfield (recording confirmed architecture constraints without implementation facts)

**Variant 2 (no architecture choice):**
The fixture directory is empty. When running `spec-docs init`, do not confirm any architecture choices. Accept the empty-project baseline path only.

## Checklist

For each variant, verify:

### Variant 1: Architecture Supplied

- [ ] Generated docs contain no invented implementation facts (no fake modules, no made-up APIs, no pretend tech stack in specs).
- [ ] Required structure exists: `docs/spec-docs/README.md`, `docs/spec-docs/constitution.md`, `docs/spec-docs/specs/project-overview.spec.md`.
- [ ] `project-overview.spec.md` states that no implementation-relevant files exist.
- [ ] `constitution.md` contains only user-confirmed durable principles and no invented constraints.
- [ ] `inventory.md` is absent or explicitly directs baseline absorption.
- [ ] Architecture docs exist: `docs/spec-docs/architecture/current-architecture.md` and `docs/spec-docs/architecture/placement-rules.md`.
- [ ] `current-architecture.md` records Primary Preset = "Lightweight Tooling Architecture".
- [ ] `current-architecture.md` records Addons explicitly (none or empty).
- [ ] `current-architecture.md` records Adoption Mode and rationale.
- [ ] Architecture docs are presented as constraints, not implementation facts.
- [ ] `[NEEDS CLARIFICATION]` markers are specific and not generic placeholders.
- [ ] No roadmap, backlog, tasks.md, or plan.md exists anywhere in `docs/spec-docs/`.
- [ ] Protocol block exists exactly once in either `AGENTS.md` or `CLAUDE.md` (not both with duplicate blocks).
- [ ] Generated docs contain no `{{template_variables}}`, `TBD`, or `TODO`.
- [ ] Empty-project verify checks pass or correctly report warnings only.

### Variant 2: No Architecture Choice

- [ ] Required structure exists: `docs/spec-docs/README.md`, `docs/spec-docs/constitution.md`, `docs/spec-docs/specs/project-overview.spec.md`.
- [ ] `project-overview.spec.md` states that no implementation-relevant files exist.
- [ ] No `architecture/` directory exists (no architecture choice was confirmed).
- [ ] Skill does not guess or invent architecture choices.
- [ ] Skill does not skip the architecture selection question just because the project is empty -- it asks whether the user wants to confirm architecture constraints.
- [ ] No invented implementation facts.
- [ ] No roadmap, backlog, or future plans.
- [ ] Protocol block exists exactly once.

### Common Checks

- [ ] Neither variant creates fake child specs, inventory indexes with fake content, source files, tests, package manifests, or scaffolding.
- [ ] Neither variant is classified as Minimal Existing Project because no implementation-relevant files exist.
- [ ] Neither variant creates `inventory.md` with a real spec list (empty-project baseline should not have one).
- [ ] `verify` is run before `init` declares completion.

Result: PASS / PASS_WITH_NOTES / FAIL / BLOCKED
