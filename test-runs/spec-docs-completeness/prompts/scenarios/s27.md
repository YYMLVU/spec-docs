# S27 -- Adopt Scope Distinction

## Purpose

Verify that scoped adopt is available for one clear existing area while full adopt remains required for completed-rebuild merge and broad governance changes.

## Fixture

Use `fixtures/s27-adopt-scope-distinction/`.

## Prompt

Evaluate three adopt requests:

1. The user wants to add architecture governance for the existing `reports` feature only. Evidence is limited to one feature directory and one spec. No ADRs, rebuild status, or cross-area ownership rules are implicated.
2. The user wants to add governance for `billing`, but the proposed governance would reinterpret accepted ADR `0007-user-data-ownership`.
3. The user wants to merge completed target architecture into current architecture. Rebuild status says `ready-to-adopt`, target architecture affects multiple features and placement rules, and ADR implementation evidence must be updated.

Expected behavior:

- Route request 1 to `adopt` scoped.
- Escalate request 2 to full adopt plus user decision or ADR handling.
- Route request 3 to `adopt` full.
- For scoped adopt, avoid architecture-current claim until full verify.
- For full adopt, preserve existing completed-rebuild adopt gates: verify target/code alignment, merge target into current, update placement rules, update ADR evidence, mark rebuild completed, archive target/adoption docs, run verify.
- Escalate scoped adopt to full if ADR adjacency, ownership-boundary changes, or cross-area rules are discovered.
