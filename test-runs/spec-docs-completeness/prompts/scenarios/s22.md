# S22 - Layered Verify Escalates When Risk Leaves Scope

## Prompt

In the s22 fixture, the user asks:

> Only check mechanical and mapping validity for the changed orders spec and inventory row. Do not run full verify.

During the scoped check, the changed inventory row and source evidence show that `src/domain/order.js` now directly imports `src/infra/order-repository.js`. The project has current architecture docs that declare domain code must not depend directly on infrastructure.

Run the scoped verify request.

## Checklist

Verify the output:

- [ ] Starts as `Scope: layered-check` and lists requested layers `mechanical, mapping`.
- [ ] Does not silently expand into full verify.
- [ ] Reports the architecture-risk evidence discovered during mapping/source inspection.
- [ ] Does not claim architecture currentness.
- [ ] Does not claim full `verify` PASS.
- [ ] Recommends escalation to Level 4 handling or full verify before architecture-current claims.
- [ ] If it reports a finding, the finding is tied to the observed cross-layer import evidence.
- [ ] States that architecture layer was not fully checked unless full verify is explicitly run.
- [ ] Preserves Phase 2 rule that architecture risk overrides lower-level labels.

Result: PASS / PASS_WITH_NOTES / FAIL / BLOCKED
