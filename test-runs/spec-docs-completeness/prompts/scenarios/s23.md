# S23 -- Scoped Placement

## Purpose

Verify that a small architecture-risk placement question uses scoped `place` instead of defaulting to broad architecture workflow.

## Fixture

Use `fixtures/s23-scoped-placement/`.

## Prompt

A project uses Feature Modular Architecture. Current architecture docs assign notification behavior to `src/features/notifications/`, and placement rules require cross-feature access through public APIs only. The user asks: "Where should I add an email notification template renderer for the existing notifications feature?"

Expected behavior:

- Classify this as architecture-relevant placement, but route to `place` scoped.
- Read only the smallest needed architecture references: current architecture and placement rules for notifications.
- Produce a placement decision with owning module, layer placement, allowed dependencies, forbidden dependencies, forbidden shortcuts, required public contracts, implementation constraints, failure localization hints, specs to read, specs to update, open questions, and confidence.
- Do not modify code.
- Do not create implementation specs as if behavior already exists.
- Do not recommend rebuild, full repair, or full adopt.
- Do not claim architecture currentness.
- Escalate only if ownership, dependency direction, public contract, or ADR relevance is unclear.
