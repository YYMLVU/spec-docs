# S10 - Hooks and Skill Collaboration

## Prompt

In the s10 fixture directory, `docs/spec-docs/` exists and has been initialized with a Feature Modular Architecture setup. The fixture also simulates the presence of a "local superpowers Spec Skill" that provides feature-level specification capabilities.

Run the following in **two variants**:

**Variant A (with local superpowers Spec Skill):**
Evaluate a non-trivial implementation change: add a "bulk export" feature that allows users to export their data as CSV. The skill should follow the workflow-integration detection order and detect the local superpowers Spec Skill presence to integrate with it.

**Variant B (without local superpowers Spec Skill):**
Evaluate the same "bulk export" feature change. The skill should fall back to Standalone Mode when no local superpowers Spec Skill is detected.

For both variants, the change evaluation should NOT actually implement code -- the goal is to observe how hooks fire, how the workflow detection and collaboration happen, and how the protocol block guides behavior.

## Checklist

Verify for both variants:

- [ ] Collaboration/fallback order is explicit in the output: first detects known workflows (Superpowers, OpenSpec, Spec-Kit), then asks about other Spec Skills, then falls back to Standalone Mode.
- [ ] Both variants preserve the workflow: `place -> implement -> update -> verify` (or the integrated equivalent).
- [ ] Protocol block guidance (from `AGENTS.md` or `CLAUDE.md`) is used -- the agent reads relevant specs and architecture docs before planning.
- [ ] Hook behavior respects reminders vs enforcement: hooks remind about required steps but do not invent blocking behavior beyond the hook policy's defined block candidates.
- [ ] No extra hook behavior is invented beyond what the hook policy (`references/hooks.md`) and skeleton hooks define.

### Variant A Specific (Integrated Mode)

- [ ] Detects that local superpowers Spec Skill is available.
- [ ] Integrates using: requirement intake -> spec-docs place -> detailed spec/plan -> implementation/TDD -> spec-docs update -> spec-docs verify.
- [ ] Spec Docs does not replace or override the local Spec Skill's planning workflow.
- [ ] Collaboration boundaries are clear: Spec Docs provides placement, facts, verification; the local Spec Skill provides feature-level specification.

### Variant B Specific (Standalone Mode)

- [ ] Detects that no local superpowers Spec Skill is available.
- [ ] Uses Standalone Mode workflow: lightweight requirement intake -> place -> minimal implementation plan -> (simulated implementation) -> update -> verify.
- [ ] Does not expand into a full requirements system, roadmap, or backlog.
- [ ] Minimal Implementation Plan is lightweight and does not replace external spec workflows.

### Hook Behavior Checks

- [ ] Hook reminders are info/warning level, not block level (unless pre-edit-guard detects a block candidate like repair mode editing business code without request).
- [ ] Hooks do not automatically modify code, create ADRs, weaken architecture rules, generate future plans, or over-block normal read-only work.

Result: PASS / PASS_WITH_NOTES / FAIL / BLOCKED
