# S28 -- Rule Ownership and Trigger Deduplication

## Scenario

The user asks the skill maintainers to reduce duplicate operational wording after Phase 1-4 are complete. The reviewer finds three stale trigger patterns:

1. Workspace template: "Before declaring work done: run `spec-docs verify`."
2. Protocol block: "Run tests. Run `spec-docs update`. Run `spec-docs verify`."
3. Hook reference: "remind update/verify around tests/builds when code changed."

## Expected Behavior Checklist

- Identifies a canonical owner for each rule family instead of spreading long rules everywhere.
- Keeps `references/modes.md` as owner for impact levels 0-4.
- Keeps `references/verification.md` as owner for targeted-check, layered-check, full-verify, and currentness gates.
- Keeps `references/hooks.md` as owner for hook behavior levels and block candidates.
- Keeps `references/project-instructions.md` plus `templates/agent-protocol-block.md` as owner/template pair for installed protocol text.
- Replaces unconditional update/verify triggers with impact-appropriate spec action wording.
- Does not require full verify for Level 0, Level 1, or Level 2 when their required action is complete.
- Preserves full verify for final init, Level 3 broad currentness/release freshness, Level 4 architecture-current claims, and repair/adopt/rebuild completion.
- Does not remove compact self-contained protocol text needed in downstream projects.
