# S07 - Rebuild State Machine

## Prompt

In the s07 fixture directory, `docs/spec-docs/` exists with current architecture defined. The project is currently a Layered Architecture. A rebuild has been initiated with the following state in `docs/spec-docs/rebuild/status.md`:

- `Current Phase: target-architecture-draft`
- Target architecture: Clean / Hexagonal Architecture with Dependency Inversion and Strong Module Boundaries addons.

Note: The `docs/spec-docs/architecture/target-architecture.md` file exists and describes the target architecture selection. The rebuild status file is the authoritative source for determining rebuild phase.

Run `spec-docs rebuild` to check or advance the rebuild state.

## Checklist

Verify the rebuild behavior:

- [ ] Reads `docs/spec-docs/rebuild/status.md` to determine the current rebuild phase.
- [ ] Determines phase from `status.md` content (`Current Phase: target-architecture-draft`), NOT the mere existence of `target-architecture.md`, to decide what actions are available.
- [ ] Does not infer phase from `target-architecture.md` existence (Hard Gate 9).
- [ ] Keeps current architecture and target architecture (`target-architecture.md`) separate and distinct.
- [ ] Treats Adoption Mode as `rebuild` for the rebuild lifecycle.
- [ ] If advancing phase (e.g., from target-architecture-draft to next phase), updates `rebuild/status.md` accordingly.
- [ ] During active rebuild, `place` and `verify` consideration of target architecture is explicitly stated.
- [ ] Does not modify current architecture rules to match target architecture prematurely.
- [ ] Rebuild progression is tracked with clear phase transitions and evidence.
- [ ] ADR rationale for the rebuild choice is referenced or confirmed.
- [ ] If the rebuild phase cannot advance (e.g., code has not been migrated yet), does not force advancement.
