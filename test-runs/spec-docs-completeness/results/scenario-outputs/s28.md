The stale trigger wording would be replaced as follows:

- The workspace README would no longer say every completion requires `spec-docs verify`; it would require the impact-appropriate spec action.
- The protocol block would no longer require universal `spec-docs update` followed by universal `spec-docs verify`; it would require impact classification and Level 0-4 completion behavior.
- The hook reference would no longer say `update/verify` around tests/builds; it would remind impact classification and impact-appropriate spec action when implementation-relevant files changed.

The skill would preserve lightweight routing:

- Level 0 is complete with a stated no-update reason.
- Level 1 is complete with the affected mapped spec update.
- Level 2 is complete with affected spec/inventory updates plus targeted light check.
- None of Level 0, Level 1, or Level 2 requires full verify merely because work is ending.

The skill would preserve strict gates:

- Final init completion still requires full included-scope verification.
- Level 3 broad currentness or release freshness still requires full verify.
- Level 4 architecture-current claims still require full verify after architecture-risk handling.
- Repair, adopt, and rebuild completion claims still require full verify.
- Architecture rules and ADRs still cannot be weakened without explicit user decision or ADR handling.

The installed protocol block may remain more self-contained than other summaries because downstream projects may only receive the copied block. That exception does not allow it to reintroduce unconditional full verify for Level 0 through Level 2.
